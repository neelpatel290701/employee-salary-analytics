# 01 — Requirements Analysis

> **Purpose of this document.** Before any line of code is written, this document restates the assessment brief in our own words, separates what is *explicitly* asked from what is *implicit* in any reasonable interpretation of it, lists the assumptions we are choosing to make, and captures the questions we would put to a product team in a real engagement. The goal is to make the reviewer confident that we understood the problem before we tried to solve it.

---

## 1. The brief, in one paragraph

We are to build a minimal but **fully usable** salary management tool for the HR Manager of an organisation with **10,000 employees**. The tool must let the HR Manager add, view, update, and delete employees through a UI, and surface salary insights — at minimum, the min/max/average salary per country and the average salary for a given job title in a country. The system must include a seed script that generates 10,000 employees by combining names from `first_names.txt` and `last_names.txt`, and the script's performance is explicitly called out. Everything must be backed by a relational database, covered by a meaningful set of fast, deterministic unit tests, and shipped as a deployed, end-to-end working application with a video demo.

The assessment is being evaluated on **how we think, design, and build**, not just on whether the application functions. Commit history, planning artifacts, and AI-tool usage are first-class evaluation criteria.

---

## 2. Explicit requirements

These are stated, verbatim or near-verbatim, in the brief.

### 2.1 Functional — Employee management
- **F1.** Add an employee via the UI.
- **F2.** View employees via the UI.
- **F3.** Update an employee via the UI.
- **F4.** Delete an employee via the UI.
- **F5.** Each employee must have at minimum: `full name`, `job title`, `country`, `salary`.
- **F6.** Additional "meaningful data" beyond the minimum is expected — the brief invites us to design what that should be.

### 2.2 Functional — Salary insights
- **F7.** Min / max / average salary per country.
- **F8.** Average salary for a given job title within a country.
- **F9.** Additional "meaningful metrics helpful for the user persona" are explicitly invited.

### 2.3 Data & seeding
- **F10.** The data layer is a relational database (the brief mentions SQLite as an example; the choice is ours — we have chosen MySQL).
- **F11.** A seed script must populate the database with **exactly 10,000 employees**.
- **F12.** Full names must be generated by combining values from `first_names.txt` and `last_names.txt`.
- **F13.** The seed script will be run **regularly** by engineers, and its **performance matters**.

### 2.4 Architecture
- **F14.** Backend and UI are both required (no UI-only or backend-only submissions).
- **F15.** UI is to be built in React or Next.js with a component library of our choice.
- **F16.** Backend framework is per JD (unspecified to us) or our choice.

### 2.5 Quality & process
- **F17.** A meaningful set of unit tests covering core functionality.
- **F18.** Tests must be **fast**, **deterministic**, and **easy to understand**.
- **F19.** Good code structure, readability, and maintainability.
- **F20.** Incremental commits showing evolution of the solution.
- **F21.** Production-quality code.
- **F22.** Intentional, documented use of AI tooling.

### 2.6 Delivery
- **F23.** A fully functional **deployed** software.
- **F24.** A **video demo**.
- **F25.** Artifacts that explain our thinking — planning notes, architecture diagrams, AI prompts, trade-offs, performance notes.

## 3. Implicit requirements

These are not stated literally in the brief, but any reasonable engineer would treat them as required. We call them out so the reviewer can see we did not skip them.

| # | Implicit requirement | Why we infer it |
|---|----------------------|-----------------|
| I1 | The insights queries must run quickly even with 10,000 employees per country | The brief explicitly seeds 10K rows; if the UI hangs, the tool is not "usable" |
| I2 | The employee list UI must remain responsive at 10K rows | Same as above — usability is part of the brief |
| I3 | Input validation on all writes (salary > 0, required fields, sane string lengths) | Production-quality code implies it; data integrity matters in HR data |
| I4 | A consistent error model so the UI can show meaningful messages | Implied by "production-quality" |
| I5 | The seed script must be repeatable (idempotent or cleanly resettable) | "Engineers run this script regularly" — they will run it more than once |
| I6 | Configuration via environment variables (DB URL, port, etc.) — no hard-coded secrets | Production-quality + deployable |
| I7 | The deployed app must actually be reachable and demonstrable | The brief asks for deployed + video |
| I8 | Names produced from the two text files should look realistic (not "John John" or all-male/all-female combinations should be possible) | "Realistic data" is implied by "10,000 employees" — toy data would defeat the insight metrics |
| I9 | Tests can be run by a reviewer with a single command | "Fast, deterministic, easy to understand" implies easy to run |
| I10 | The codebase is readable without a guided tour | "Readability and maintainability" are explicit |

## 4. Non-functional requirements

| # | Requirement | Target |
|---|-------------|--------|
| N1 | Seed script runtime for 10,000 employees | Under ~2 seconds on a modern laptop (specific number to be benchmarked and documented in [`07-performance-plan.md`](07-performance-plan.md)) |
| N2 | Employee-list API response time at 10K rows | < 200 ms p95 for a single page of results |
| N3 | Insights API response time at 10K rows | < 300 ms p95 for any single metric query |
| N4 | Unit-test suite duration | Under ~5 seconds for the full suite |
| N5 | All tests deterministic | No flakes; no reliance on real wall clock, random seeds without fixed seed, or network |
| N6 | Code coverage on business logic | "Meaningful" — we are not chasing a number, but the insights aggregations and request handlers should be well-covered |

## 5. Assumptions

These are conscious decisions we are making in the absence of a product owner to ask. Each is recorded so it can be challenged.

| # | Assumption | Rationale |
|---|------------|-----------|
| A1 | Salaries are stored as a single numeric value in a single currency (USD), already-normalised at write time | Multi-currency comparison is non-trivial (FX rates, historical rates, display preferences). We will flag this clearly as a known simplification rather than implement it half-way. |
| A2 | Country is stored as an ISO 3166-1 alpha-2 code (e.g. `US`, `IN`, `DE`), and rendered as a human-readable name in the UI | Avoids "United States" vs "USA" vs "US" data-quality issues that would silently break the country-level aggregations |
| A3 | Job title is a free-form string in the data model, but the UI offers an autocomplete from existing values to encourage consistency | The brief does not give us a controlled taxonomy. Free-form preserves flexibility; autocomplete protects analytics. |
| A4 | The HR Manager is a single trusted user. No authentication is implemented. | The brief does not mention auth. Adding it would be scope creep that distracts from what is being evaluated. We will call this out in [`08-tradeoffs.md`](08-tradeoffs.md). |
| A5 | Deletion is a hard delete | Soft-delete with a `deletedAt` column would be the right move in production; for this assessment, hard delete is simpler and the brief does not ask for restore/audit. We document the trade-off rather than over-build. |
| A6 | The seed script wipes existing employees before inserting new ones | Engineers run it "regularly" — keeping leftover rows would corrupt analytics. We will guard this with an explicit prompt or env flag so it cannot happen by accident in any environment that resembles production. |
| A7 | Email is unique per employee | Standard HR-system convention; needed if we expose employee detail by a stable identifier |
| A8 | Salary represents **annual gross compensation** in the chosen currency | Removes ambiguity in aggregations; documented in the UI |
| A9 | The 10,000-employee data set is small enough to live on a single MySQL instance with no sharding | True for this volume; we are not going to over-engineer for hypothetical scale |
| A10 | The reviewer can run Docker on their machine | We use Docker Compose for the local MySQL. If this is an issue, we will provide a fallback in the README. |

## 6. Out of scope

We are explicitly **not** building the following, even though they would be relevant in a real production HR tool. Stating them here protects against scope creep and shows the reviewer that we are making deliberate choices.

- Authentication, authorisation, role-based access control.
- Multi-tenancy (multiple organisations).
- Audit logs / change history.
- Soft deletes and restore.
- CSV import / export (beyond the seed script).
- Email notifications, alerts, scheduled reports.
- Internationalisation / localisation of UI strings (English only).
- Multi-currency display with FX conversion.
- Salary history (we model **current** salary only; historical salary trend would be a natural extension).
- Mobile-specific UI (responsive desktop is acceptable; native mobile is not).
- PII redaction / GDPR tooling beyond the implicit data minimisation already in our model.
- Real-time updates / websockets / presence indicators.

## 7. Open questions

In a real engagement we would put these to the product team before starting. For the purpose of the assessment, each has been resolved by the assumption noted in the right-hand column.

| # | Open question | Resolution we are running with |
|---|---------------|--------------------------------|
| Q1 | How should salary be normalised across countries — same currency, or local currency? | Assumption A1: single normalised currency (USD) |
| Q2 | Are deletions reversible? | Assumption A5: hard delete |
| Q3 | How "fresh" should the seed be — preserved between runs, or wiped each time? | Assumption A6: wiped, with a safety guard |
| Q4 | Should the HR Manager see only their direct reports, or the whole organisation? | The brief implies the whole organisation. No filtering by reporting line. |
| Q5 | Is there a regulatory constraint on storing salary data that should affect schema design (e.g. encryption at rest, separation of identity from salary)? | Out of scope per Section 6; would be addressed in a real engagement. |
| Q6 | Should the UI support bulk operations (delete many, edit many)? | Out of scope. Not asked for. |
| Q7 | Do we need a "history" of changes to an employee record? | Out of scope per Section 6. Documented as a natural follow-on. |

## 8. Success criteria

The assessment is "done" when **all** of the following are true:

1. Every explicit requirement (F1–F25) is implemented and demonstrably works in the deployed application.
2. Every non-functional target (N1–N6) is met or exceeded, with the seed-script and query benchmarks recorded in [`07-performance-plan.md`](07-performance-plan.md).
3. The commit history reads as a clear narrative: planning → tests-first → implementation → refactor, with no "kitchen-sink" commits.
4. The `docs/` directory tells the full design story without the reviewer needing to read the code first.
5. AI-tool usage is honestly and specifically documented in [`09-ai-usage.md`](09-ai-usage.md) — including where AI was over-ruled.
6. A reviewer can clone the repo, run `docker compose up -d` plus two `npm` commands, and have a working application within five minutes.
7. The video demo walks through every requirement in under five minutes.
